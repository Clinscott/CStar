# Progression and Manifestation Doctrine

Status: derived canon from CStar White Paper v0.3

## Separate Ladders

CStar progression is not one ladder. The following dimensions must remain
separate:

- Initiation track: basic CStar setup and first working loop.
- Operator level: long-term operator mastery.
- Spoke mastery: maturity and reliability of a specific spoke.
- Build rank: coherence of the active loadout.
- Manifestation mastery: quality and privacy posture of visual identity.
- Guild renown: community trust, adoption, mentorship, and contribution.
- Warden seals: validation, safety, documentation, and provenance signals.

Setup completion ends at Awakened. It does not imply Ascension.

## Initiation Track

The initiation track runs from 0 to 10 and proves the basic CStar loop:

0. Unbound: concept exists, no bound working system.
1. Spark: project initialized.
2. Core Online: Star Core or hub is running.
3. First Spoke: at least one spoke is registered.
4. First Cast: first safe action executed.
5. First Quest: first Bead created.
6. First Warden: first validation or security check connected.
7. First Engram: first memory or lore entry recorded.
8. First Loadout: profile and equipped spokes visible.
9. First Ritual Draft: repeatable workflow defined but not necessarily
   autonomous.
10. Awakened: basic loop exists end to end.

## Utility Dimensions

Utility is contextual. CStar should track these dimensions separately:

- Personal affinity: operator-relative meaning and enjoyment.
- Build synergy: fit with the current loadout, goals, and style.
- Validated performance: evidence that the spoke does what it claims.
- Community renown: optional ecosystem signal from reviews, audits, and
  adoption.

No single global usefulness score should flatten these dimensions.

## Manifestation Doctrine

Each spoke may have three forms:

- Functional form: what it actually does.
- Mythic form: what it means symbolically.
- Manifest form: how the operator sees it.

Changing manifestation does not change functional authority. Improving
functional contract does not force a specific manifestation.

## Visibility Layers

| Layer | May show | Must not show |
| --- | --- | --- |
| Public | Class, build archetype, public manifestations, cosmetic relics, selected badges. | Live security posture, private spokes, device identity, exact infrastructure, ritual schedules. |
| Trusted peer | Selected collaboration state, guild membership, chosen spokes, verified public seals. | Secrets, private memory, sensitive automation, non-consented data. |
| Private operator | Full manifestation detail, capability maps, hidden spokes, logs, tasks, authority states. | N/A; private view remains local/operator-controlled. |

## Manifestation Rule

Manifestation is identity, not evidence of authority. A spoke can look like a
divine sword and still only have observe authority.
