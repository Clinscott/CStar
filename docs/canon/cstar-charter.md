# CStar Charter

Status: derived canon from CStar White Paper v0.3

## Purpose

CStar is a sovereign, game-native AI operating framework. It gives an operator
a deterministic Star Core, an extensible set of spokes, a loadout model, a
validation and memory loop, and an optional manifestation layer for identity and
presence.

CStar is not only a chatbot, automation dashboard, or agent marketplace. It is
a personal AI operating layer where operators create, equip, validate, and
evolve bounded AI modules through explicit authority and proof.

## Product Thesis

CStar succeeds when it makes serious capability feel magical without allowing
the metaphor to hide the mechanism.

The operator-facing layer may use game language, mythic language, and avatar
language. The system layer must preserve precise engineering language for
authority, execution, validation, privacy, and risk.

## Non-Negotiable Rules

- No universal usefulness score. Utility is contextual, operator-relative, and
  goal-dependent.
- Visual form never grants authority.
- Roadmap phase is not operator level.
- Setup is not Ascension.
- Personal affinity, validated performance, and community renown are separate
  dimensions.
- Mythic terms must map to real system functions.
- High-authority actions require explicit policy, validation, logs, and
  operator control.
- CStar may propose, draft, and organize. It may not silently expand its own
  authority.
- AR and public presence must express identity without leaking sensitive
  operational detail.

## System Shape

CStar has these primary layers:

- Operator / Magus: the sovereign human authority.
- Mobile Sanctum / Console: the primary command surface.
- Star Core / Kernel: deterministic registry, routing, state, manifest,
  policy, and authority layer.
- Grimoire / Spellbook: human-facing capability catalog and action semantics.
- Spoke Registry: installed, equipped, active, deprecated, or quarantined
  spokes.
- Quest Log / Beads: scoped work units with criteria, ownership, and
  validation.
- Engrams / Hall: recorded outcomes, decisions, lessons, and memories.
- Wardens: validation, safety, quality, policy, and security gates.
- Forge: controlled self-building and community spoke creation.
- Aura: privacy-filtered profile, avatar, manifestation, sigil, relic, and AR
  projection layer.

## Refinement Loop

The canonical CStar loop is:

1. Refine.
2. Ratify.
3. Implement.
4. Validate.
5. Engrave.
6. Progress.

No major phase, spoke, ritual, public feature, authority increase, or
self-building workflow should skip the refinement gate. Acceptance is not
execution, and execution is not publication.

## Current Implementation Posture

This charter is canonical guidance, not runtime migration by itself. Runtime
schemas, UI surfaces, and spoke registry changes must be handled through
separate gated implementation PRs.
