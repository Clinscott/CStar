# Fallows Hallow RPG - Game Design & Technical Specification

## 1. Core Concept
An epic, 16-bit JRPG inspired by FFIV and FFVI, built on a custom engine. It features a sweeping narrative, ensemble cast, active time battles (ATB), and dynamic storytelling driven by a local LLM (Gemma 4) running the Taliesin Phoenix Loop. The player follows the fixed, canonical "rails" of the Fallows Hallow storyline, but the specific flavor, dialogue, and party interactions are generated dynamically based on player choices and party composition.

## 2. Technical Architecture
*   **Engine:** Custom-built from scratch. No bloated physics engines (Unity/Godot) or general-purpose frameworks (Phaser).
*   **Language:** TypeScript (for strict typing of RPG databases, stats, and states).
*   **Renderer:** PixiJS (acts solely as a hyper-fast 2D WebGL canvas renderer).
*   **AI Integration:** Local API bridge to Gemma 4 running the Taliesin v3.0 skill. The game client sends JSON state payloads (party, location, inventory); Taliesin returns structured JSON dialogue/narration.
*   **Architecture Pattern:** Entity-Component-System (ECS) to handle grid movement, ATB combat logic, and UI rendering efficiently.

## 3. Art & Audio Pipeline
*   **Visual Style:** 16-bit SNES era (FFVI standard). Pixel art sprites with rich, atmospheric palettes matching the `CORE_RULES.md` (e.g., burning autumn trees, deep shadows).
*   **Sprite Creation:** Base keyframes (Idle, Walk, Attack, Hurt, Cast) drawn manually (e.g., in Aseprite).
*   **VFX:** Custom programmatic particle systems built in the engine for magic and summons (screen shakes, color blending, Mode-7 style scaling) to create massive spectacle without needing to hand-animate every frame of a spell.
*   **UI:** Clean, high-contrast menus with classic JRPG typography.

## 4. Gameplay Mechanics

### 4.1 Exploration
*   **Movement:** Crisp, grid-based tile movement. 
*   **Environments:** Broad sweeping landscapes (overworld map) and dense, detailed dungeons (The Woods, The Haunted House).
*   **Interactions:** Dynamic NPC dialogue and environmental observations generated by Taliesin based on the active party leader.

### 4.2 Combat (ATB System)
*   **Flow:** Active Time Battle. Tension is created by managing cooldowns while enemies (e.g., the massive wolf, Mathias's constructs) attack in real-time.
*   **Party Setup:** Ensemble cast. Players can swap party members, which alters combat strategy and triggers unique dynamic dialogue.

### 4.3 Character Classes & Skills (Draft)
*   **John (The Seeker):** Balanced frontline. Skills focus on protective strikes and uncovering truth (perhaps unlocking temporal skills later via his link to Rowan).
*   **Gideon (The Survivor):** Rogue/Thief archetype. High evasion, desperate strikes. Skills involve stealing food or items to survive.
*   **Nicci (The Guardian):** Defender/Wild Mage. Protective buffs and eventual affinity with wild, ancient magic (nature/wolf connection).
*   **Alfred (The Scholar):** Analyst/Mage. Uses logic and books to Scan enemies, identifying weaknesses and structural flaws.
*   **The Twins (Amanda & Brittany):** Dual-caster archetype. Must act in sync for massive damage, or support the group with unified buffs.

## 5. Narrative Integration (Taliesin)
*   **The Rails:** The main story beats (The Inciting Incident in the Fire Room, Starving in the Woods, The Road to the White Spot, Confronting Mathias) are hardcoded.
*   **The Dynamic Master:** Taliesin generates the dialogue between these beats.
*   **Context Safety:** The game engine enforces `Canonical Logic & Inventory` by only passing the player's *actual* inventory to Taliesin, preventing the AI from hallucinating solutions (like matches).
*   **Structured Output:** The engine requires Gemma 4 to output strict JSON (e.g., `{"speaker": "Mathias", "text": "...", "emotion": "grimace"}`) to drive the game's portrait and text box system.
